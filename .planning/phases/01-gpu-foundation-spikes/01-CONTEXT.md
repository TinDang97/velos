# Phase 1: GPU Foundation & Spikes - Context

**Gathered:** 2026-03-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Validate that the wgpu/Metal compute pipeline works on Apple Silicon: fixed-point arithmetic with emulated i64, wave-front dispatch ordering, ECS-to-GPU buffer round-trip, per-lane leader index with dual-leader tracking, and deterministic PCG hash. Pure technical validation with 1K synthetic agents — no road network, no real physics, no simulation logic.

</domain>

<decisions>
## Implementation Decisions

### Fixed-point overflow strategy
- Use emulated i64 in WGSL (hi/lo i32 split) to preserve Q16.16 as designed
- Implement emulated division as well (shift-before-divide via emulated i64)
- All three formats must be verified: Q16.16 (position), Q12.20 (speed), Q8.8 (lateral offset)
- Edge cases required: overflow boundaries, negative values, zero/near-zero, round-trip fidelity (f64 -> fixed -> WGSL -> fixed -> f64 within 1 LSB)
- CPU-GPU sync verified via golden test vectors generated by a Python script (checked into repo under test fixtures)

### Spike pass/fail criteria
- 1K test agents for ECS-to-GPU round-trip
- Frame time target: < 16ms (60 FPS) on Apple Silicon
- Per-lane leader index tested with synthetic data (no real roads): 20 lanes x 50 agents
- PCG hash: determinism (same input = same output across runs) + basic uniformity (chi-squared on 10K samples)
- Wave-front correctness: CPU reference comparison AND invariant assertions (no overtaking, gaps >= 0, monotonic positions)

### Workspace bootstrapping
- Create only velos-core + velos-gpu (no empty scaffolds for other crates)
- Local-only quality gates (no CI in Phase 1)
- Nightly Rust toolchain pinned via rust-toolchain.toml
- Workspace-level dependency declarations in root Cargo.toml
- MIT license
- main + feature branches (simple trunk-based)
- Minimal README.md (name, description, build/test commands, link to docs/architect/)
- Default rustfmt + clippy config (no custom files)

### Crate boundaries
- Fixed-point types (FixedQ16_16, FixedQ12_20, FixedQ8_8) live in velos-core
- ECS component structs (Position, Kinematics) defined in velos-core
- velos-gpu owns GPU buffer layout (how components map to SoA buffers)
- velos-gpu exposes high-level API (ComputeDispatcher, BufferPool, ShaderRegistry) — no raw wgpu leakage
- Per-crate error types with thiserror (#[from] wrapping)

### Test organization
- GPU integration tests live in velos-gpu crate (tests/ directory)
- GPU tests gated by feature flag AND runtime skip (feature = "gpu-tests" + wgpu::Instance adapter check)
- Fixed-point tests: example-based + golden vectors (no property-based testing)
- Test naming: descriptive style (test_fix_mul_overflow_boundary_produces_correct_result)

### Benchmark infrastructure
- Built-in #[bench] harness (nightly)
- Four metrics: GPU dispatch time, buffer readback time, full round-trip time, agents per second
- Results written to JSON baseline file (benchmarks/baseline.json) for regression comparison

### Wave-front dispatch
- Synthetic lane arrays (no graph structure) — 20 lanes x 50 agents
- Compute shader does leader-follower gap update to prove ordering
- Verification: CPU reference comparison + invariant assertions (both)
- Correctness: if dispatch order is wrong, agents read stale leader data -> detectable mismatch vs CPU reference

### ECS-to-GPU round-trip
- Dense index array maps GPU index -> ECS entity (rebuild on spawn/despawn)
- Double buffering from the start (front/back GPU buffers, swap per frame)
- Round-trip spike packs Position + Kinematics components

### Dual-leader lane-change tracking
- LaneChangeState component: { source_lane, target_lane, progress: f32 }
- Must prove: both leaders found correctly, gap respects closer (more conservative) leader, clean transition at progress=1.0, handles empty target lane

### Spike ordering
- Parallel where possible:
  - Track A: fixed-point types (CPU-only, velos-core)
  - Track B: wgpu device + buffers (GPU setup, velos-gpu)
  - Merge -> Track C: ECS round-trip (requires A+B)
  - **Go/no-go gate after Track C** — if round-trip fails, stop and redesign
  - Track D: wave-front dispatch + leader sort (requires C)
  - Track E: PCG hash (independent, can parallel with D)
  - Final: integrated benchmark

### Documentation
- Inline code docs: standard /// with examples, # Safety for unsafe
- Benchmark results: stdout + JSON baseline file
- No separate spike summary docs — code is the documentation

### Claude's Discretion
- Exact emulated i64 implementation details (hi/lo split algorithm)
- GPU buffer stride and alignment choices
- Internal module organization within each crate
- Exact Python script structure for golden vector generation
- wgpu adapter selection and device configuration
- Benchmark iteration counts and warm-up strategy

</decisions>

<specifics>
## Specific Ideas

- WGSL i64 emulation via hi/lo i32 split is the critical technique — if this doesn't work cleanly, the entire fixed-point strategy needs rethinking
- Go/no-go gate after ECS round-trip (Track C) prevents wasted effort on wave-front/PCG if the fundamental pipeline is broken
- "Leader-follower gap update" as wave-front test kernel — directly models the real simulation pattern (IDM reads leader position) rather than an abstract test
- Dual-leader tracking uses conservative gap (smaller of two) to prevent collision during lane-change transitions

</specifics>
