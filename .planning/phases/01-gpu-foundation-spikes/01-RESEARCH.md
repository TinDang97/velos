# Phase 1: GPU Foundation & Spikes - Research

**Researched:** 2026-03-06
**Domain:** wgpu GPU compute, fixed-point arithmetic, ECS-GPU buffer interop, WGSL shader development
**Confidence:** HIGH

## Summary

Phase 1 bootstraps the VELOS Rust workspace (velos-core + velos-gpu) and validates the GPU compute pipeline on Apple Silicon/Metal. The critical technical risk is WGSL's lack of native i64 -- fixed-point Q16.16 multiplication requires 64-bit intermediates, which must be emulated using hi/lo i32 pairs. This emulation technique is well-documented and has working implementations (Gold18K/WebGPU-WGSL-64bit-BigInt library provides Karatsuba-based multiplication in pure WGSL).

The wgpu ecosystem is mature (v28.0.0 as of early 2026) with solid Metal backend support on Apple Silicon. hecs (v0.10.5) is a minimal ECS with columnar SoA storage ideal for GPU buffer projection. PCG hash is the standard approach for deterministic GPU random numbers (Jarzynski & Olano, JCGT 2020). The nightly `#[bench]` harness is still unstable -- criterion 0.8 is the stable alternative, but the CONTEXT.md locks in nightly `#[bench]`, which is acceptable since the project already requires nightly for other reasons.

**Primary recommendation:** Start with Track A (fixed-point CPU types in velos-core) and Track B (wgpu device/buffer setup in velos-gpu) in parallel, merge into Track C (ECS-GPU round-trip), gate on Track C success before proceeding to wave-front dispatch and PCG hash.

### Pattern 1: Double-Buffered GPU Buffer Pool
**What:** Two GPU storage buffers per component type; GPU reads front buffer while CPU prepares back buffer; swap each frame.
**When to use:** Always for GPU-resident component data to prevent read-write hazards.
**Example:**
```rust
// Source: docs/architect/01-simulation-engine.md Section 5
pub struct BufferPair {
    front: wgpu::Buffer,  // GPU reads during compute dispatch
    back: wgpu::Buffer,   // CPU writes / GPU writes results
}

impl BufferPair {
    pub fn new(device: &wgpu::Device, size: u64, label: &str) -> Self {
        let create = |suffix| device.create_buffer(&wgpu::BufferDescriptor {
            label: Some(&format!("{label}_{suffix}")),
            size,
            usage: wgpu::BufferUsages::STORAGE
                 | wgpu::BufferUsages::COPY_SRC
                 | wgpu::BufferUsages::COPY_DST,
            mapped_at_creation: false,
        });
        Self { front: create("front"), back: create("back") }
    }

    pub fn swap(&mut self) {
        std::mem::swap(&mut self.front, &mut self.back);
    }
}
```

### Pattern 2: GPU Readback via Staging Buffer
**What:** Storage buffers cannot be directly mapped for CPU read. Copy to a MAP_READ staging buffer, then map it.
**When to use:** Every time you need to read GPU results back to CPU (test verification, benchmark measurement).
**Example:**
```rust
// Source: wgpu hello-compute example + official docs
let staging_buffer = device.create_buffer(&wgpu::BufferDescriptor {
    label: Some("staging"),
    size: data_size,
    usage: wgpu::BufferUsages::MAP_READ | wgpu::BufferUsages::COPY_DST,
    mapped_at_creation: false,
});

// After compute dispatch, copy storage -> staging
encoder.copy_buffer_to_buffer(&storage_buffer, 0, &staging_buffer, 0, data_size);
queue.submit(Some(encoder.finish()));

// Map staging buffer for CPU read
let buffer_slice = staging_buffer.slice(..);
let (sender, receiver) = flume::bounded(1);
buffer_slice.map_async(wgpu::MapMode::Read, move |result| {
    sender.send(result).unwrap();
});
device.poll(wgpu::Maintain::Wait);
receiver.recv().unwrap().unwrap();

let data = buffer_slice.get_mapped_range();
let result: &[u8] = &data;
// Cast back using bytemuck::cast_slice
```

### Pattern 3: Emulated i64 in WGSL (hi/lo i32 split)
**What:** WGSL has no native i64. Represent 64-bit integers as a pair of i32 values (high 32 bits, low 32 bits). Implement add, subtract, multiply, and divide using schoolbook algorithms.
**When to use:** Fixed-point multiplication where Q16.16 * Q16.16 produces a 64-bit intermediate before right-shifting back to 32-bit result.
**Example:**
```wgsl
// Emulated i64 as struct
struct I64 {
    hi: i32,  // high 32 bits (signed)
    lo: u32,  // low 32 bits (unsigned, to avoid sign-extension issues)
}

// Multiply two i32 values, producing emulated i64 result
fn mul_i32_to_i64(a: i32, b: i32) -> I64 {
    // Split into 16-bit halves to avoid i32 overflow
    let a_lo = u32(a) & 0xFFFFu;
    let a_hi = u32(a) >> 16u;
    let b_lo = u32(b) & 0xFFFFu;
    let b_hi = u32(b) >> 16u;

    let ll = a_lo * b_lo;
    let lh = a_lo * b_hi;
    let hl = a_hi * b_lo;
    let hh = a_hi * b_hi;

    let mid = lh + hl;  // may overflow, handled below
    let carry = select(0u, 1u, mid < lh);  // overflow carry

    let lo = ll + (mid << 16u);
    let lo_carry = select(0u, 1u, lo < ll);

    let hi = hh + (mid >> 16u) + (carry << 16u) + lo_carry;

    // Handle signs (XOR sign bits of a and b)
    // ... sign correction for negative inputs
    return I64(i32(hi), lo);
}

// Fixed-point Q16.16 multiply: (a * b) >> 16
fn fix_mul_q16_16(a: i32, b: i32) -> i32 {
    let product = mul_i32_to_i64(a, b);
    // Right-shift the 64-bit result by 16
    return i32((product.lo >> 16u) | (u32(product.hi) << 16u));
}
```

### Pattern 4: PCG Hash in WGSL
**What:** Permuted Congruential Generator hash for deterministic pseudo-random numbers on GPU.
**When to use:** Any place needing reproducible random noise (IDM driver imperfection, lane-change stochasticity).
**Example:**
```wgsl
// Source: Jarzynski & Olano, "Hash Functions for GPU Rendering", JCGT 2020
// + Nathan Reed's blog adaptation
fn pcg_hash(input: u32) -> u32 {
    let state = input * 747796405u + 2891336453u;
    let word = ((state >> ((state >> 28u) + 4u)) ^ state) * 277803737u;
    return (word >> 22u) ^ word;
}

// For (agent_id, step) -> deterministic noise
fn agent_random(agent_id: u32, step: u32) -> f32 {
    let seed = agent_id ^ (step * 1664525u + 1013904223u);
    let hash = pcg_hash(seed);
    return f32(hash) / 4294967295.0;  // normalize to [0, 1)
}
```

### Pattern 5: Dense Index Map (ECS <-> GPU)
**What:** A contiguous Vec<Entity> where position i corresponds to GPU buffer index i. Rebuilt on spawn/despawn.
**When to use:** Always, for mapping hecs Entity handles to GPU array indices.
**Example:**
```rust
pub struct IndexMap {
    /// GPU index -> ECS entity
    gpu_to_entity: Vec<hecs::Entity>,
    /// ECS entity -> GPU index (sparse, using HashMap for now)
    entity_to_gpu: HashMap<hecs::Entity, u32>,
}

impl IndexMap {
    pub fn rebuild(&mut self, world: &hecs::World) {
        self.gpu_to_entity.clear();
        self.entity_to_gpu.clear();
        for (entity, _) in world.query::<&Position>().iter() {
            let idx = self.gpu_to_entity.len() as u32;
            self.gpu_to_entity.push(entity);
            self.entity_to_gpu.insert(entity, idx);
        }
    }
}
```

### Anti-Patterns to Avoid
- **Forgetting COPY_SRC on storage buffers:** wgpu panics at runtime if you try to copy from a buffer without COPY_SRC usage flag. Always include COPY_SRC on any buffer you need to read back.
- **Mapping storage buffers directly:** MAPPABLE_PRIMARY_BUFFERS is not universally supported. Always use a separate staging buffer with MAP_READ | COPY_DST for readback.
- **Forgetting device.poll() after map_async:** On native (non-WASM) targets, the map callback never fires unless you poll the device. Always call `device.poll(wgpu::Maintain::Wait)` after `map_async`.
- **Using pow() in WGSL:** `pow(x, 4.0)` has undefined behavior for negative x. Use `safe_pow4(x) = x*x*x*x` instead.
- **Signed right-shift in WGSL i32:** WGSL arithmetic right-shift preserves the sign bit, which is correct for fixed-point. But mixing signed and unsigned bit operations is error-prone. Use u32 for bit manipulation, i32 for arithmetic.
- **Exceeding workgroup invocation limit on Metal:** Apple Silicon caps total workgroup invocations at 256 (not per-dimension, but total product). `@workgroup_size(256, 1, 1)` is fine; `@workgroup_size(256, 2, 1)` is NOT.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Pod/Zeroable trait impls | Unsafe manual impls | bytemuck derive macros | Derive validates padding, alignment, and field types at compile time |
| Byte slice casting | `unsafe { std::slice::from_raw_parts }` | `bytemuck::cast_slice` | Type-safe, checked at compile time |
| WGSL shader validation | Runtime-only testing | naga-cli `naga --validate` | Catches syntax and type errors without GPU; runs in CI |
| Async GPU poll | Custom future/executor | pollster::block_on | Dead-simple sync wrapper for wgpu async in tests |
| Error types | String-based errors | thiserror derive | Typed errors with #[from] for composability |

**Key insight:** The GPU compute domain has specific footguns (buffer usage flags, alignment, async mapping) that established libraries handle correctly. Bytemuck in particular prevents entire categories of UB that manual transmutation invites.

## Common Pitfalls

### Pitfall 1: WGSL i32 Overflow in Fixed-Point Multiplication
**What goes wrong:** Q16.16 * Q16.16 overflows i32. For example, 1000.0 * 1000.0 in Q16.16: (1000 << 16) * (1000 << 16) = 65,536,000 * 65,536,000 -- far beyond i32 range.
**Why it happens:** WGSL has no native i64, and i32 multiplication wraps silently.
**How to avoid:** Split each operand into 16-bit halves, multiply the four partial products separately, then reassemble. This is the hi/lo i32 emulation technique mandated by CONTEXT.md.
**Warning signs:** Fixed-point test vectors fail for large values but pass for small values.

### Pitfall 2: Buffer Alignment Padding
**What goes wrong:** GPU reads garbage data because struct fields aren't aligned to GPU expectations.
**Why it happens:** WGSL structs have specific alignment rules (vec4 -> 16-byte aligned, f32 -> 4-byte aligned). Rust #[repr(C)] structs may not match.
**How to avoid:** Use `#[repr(C)]` on all GPU-mapped structs. Add explicit padding fields if needed. Write a test that asserts `std::mem::size_of::<T>()` matches the expected GPU stride. WGSL storage buffers require struct alignment of 16 bytes for vec4, 8 bytes for vec2.
**Warning signs:** GPU results are correct for the first few agents but corrupted for later ones (stride mismatch).

### Pitfall 3: Workgroup Size vs Agent Count Mismatch
**What goes wrong:** Dispatch too few workgroups and some agents don't get updated. Dispatch too many and the shader reads out-of-bounds.
**Why it happens:** `dispatch_workgroups(n)` launches `n * workgroup_size` invocations. Must ceil-divide agent count by workgroup size.
**How to avoid:** Always use `dispatch_workgroups((agent_count + 255) / 256)` for workgroup_size 256. Always guard in shader: `if (gid.x >= params.agent_count) { return; }`.
**Warning signs:** Works for exact multiples of 256 but fails for other counts.

### Pitfall 4: Nightly #[bench] Instability
**What goes wrong:** `#[bench]` is unstable and may change between nightly releases.
**Why it happens:** The built-in bench harness has been unstable since 2015 with no stabilization timeline.
**How to avoid:** Pin a specific nightly version in rust-toolchain.toml. If `#[bench]` breaks in a future nightly, criterion 0.8 is a drop-in replacement that works on stable.
**Warning signs:** `cargo bench` stops compiling after a rustup update.

### Pitfall 5: Metal-Specific Workgroup Storage Limit
**What goes wrong:** Compute shader compiles but crashes or produces wrong results on Metal.
**Why it happens:** Apple Silicon has 16,384 bytes max workgroup storage (shared memory). If your shader uses `var<workgroup>` arrays exceeding this, Metal rejects it silently.
**How to avoid:** Keep workgroup-shared arrays small. For 256 agents per workgroup with 12-byte Position structs: 256 * 12 = 3,072 bytes -- well within limit. But if you store multiple arrays (position + kinematics + leader indices), sum all of them.
**Warning signs:** Shader works on Vulkan but fails on Metal. Or: works with small workgroup sizes but fails with 256.

### Pitfall 6: PCG Hash Uniformity Depends on Seed Quality
**What goes wrong:** PCG output has visible patterns or correlations between adjacent agent IDs.
**Why it happens:** If `agent_id` values are sequential (0, 1, 2...) and step is constant, adjacent seeds differ by 1, which may not produce visually independent outputs from a single PCG round.
**How to avoid:** Mix agent_id and step before feeding to PCG: `seed = agent_id ^ (step * 1664525u + 1013904223u)`. This uses an LCG-style scramble to spread seed bits. Verify with chi-squared test on 10K samples as specified in CONTEXT.md.
**Warning signs:** Chi-squared test fails, or visual inspection of noise shows bands/stripes.

### hecs World to GPU Buffer Projection
```rust
// Source: hecs docs + architecture pattern
pub fn project_to_gpu(
    world: &hecs::World,
    queue: &wgpu::Queue,
    position_buffer: &wgpu::Buffer,
    kinematics_buffer: &wgpu::Buffer,
    index_map: &IndexMap,
) {
    let mut positions = Vec::with_capacity(index_map.len());
    let mut kinematics = Vec::with_capacity(index_map.len());

    for entity in index_map.entities() {
        let pos = world.get::<&Position>(entity).unwrap();
        let kin = world.get::<&Kinematics>(entity).unwrap();
        positions.push(GpuPosition::from(*pos));
        kinematics.push(GpuKinematics::from(*kin));
    }

    queue.write_buffer(position_buffer, 0, bytemuck::cast_slice(&positions));
    queue.write_buffer(kinematics_buffer, 0, bytemuck::cast_slice(&kinematics));
}
```

|--------------|------------------|--------------|--------|
| wgpu 0.x API (Adapter::request_device sync) | wgpu 28.x (fully async device creation) | 2024-2025 | Must use async/await or pollster for device init |
| WGSL `@stage(compute)` | WGSL `@compute` | WGSL spec 2023 | Old attribute syntax removed; use `@compute` |
| `wgpu::BufferUsage` (singular) | `wgpu::BufferUsages` (plural) | wgpu 0.12+ | Renamed to `BufferUsages`, `TextureUsages`, etc. |
| `queue.write_buffer` limited size | Chunked writes for large buffers | Ongoing | For >128MB buffers, may need multiple writes; 1K agents is tiny so not a concern |
| naga as separate crate | naga integrated in wgpu | wgpu 22+ | naga-cli still installable separately for validation |

**Deprecated/outdated:**
- `wgpu::Backends::PRIMARY`: Replaced by explicit backend selection or `wgpu::Backends::all()`
- `adapter.get_info()`: Changed to `adapter.get_info()` returning `AdapterInfo` struct (field names changed across versions)
- `#[stage(compute)]` attribute in WGSL: Replaced by `@compute`

## Open Questions

1. **Exact wgpu 28.0 API surface for device initialization**
   - What we know: The async API pattern is well-established; Metal backend works on Apple Silicon
   - What's unclear: Exact method signatures in v28 may differ from earlier versions in examples
   - Recommendation: Pin wgpu = "28.0" in Cargo.toml and let compiler errors guide API updates. The hello-compute example in the wgpu repo is always updated to latest.

2. **nightly #[bench] stability window**
   - What we know: The feature has been unstable since 2015 with no RFC for stabilization
   - What's unclear: Whether a specific nightly pin will remain compatible for the project duration
   - Recommendation: Pin nightly date in rust-toolchain.toml. If it breaks, switching to criterion 0.8 is a 1-hour migration.

3. **Emulated i64 division performance in WGSL**
   - What we know: Multiplication via hi/lo split is well-documented. Division is harder and slower.
   - What's unclear: Whether emulated division is fast enough for per-agent-per-frame computation
   - Recommendation: Implement division but benchmark it. If too slow, consider shift-only approximations where division by power-of-2 is sufficient (which it often is for fixed-point scaling).
