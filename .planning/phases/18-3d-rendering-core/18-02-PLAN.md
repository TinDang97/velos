<objective>
Generate 3D road surface polygons and lane markings from the velos-net RoadGraph, with junction surface fills. Upload as static GPU buffers rendered by Renderer3D.

Purpose: Roads are the primary visual reference in 3D view. Without road surfaces, the scene is just a ground plane. This plan generates all road geometry at load time from existing RoadGraph data, requiring no new data sources.

Output: Textured road surfaces (flat grey), white lane markings (dashed center, solid edge), lighter grey junction surfaces -- all rendered with depth-correct 3D perspective.
</objective>

  <behavior>
    - Test: generate_road_mesh for a straight 2-point edge with lane_count=2 produces 6 vertices (2 triangles) with correct width (2 * 3.5m = 7m total)
    - Test: road vertices are at y=0.0 (ground level) with 3D coords (x, 0, y_2d)
    - Test: generate_lane_markings for a 2-lane edge produces center dashed line vertices at y=+0.01 and two solid edge lines
    - Test: center line dashes are 3m line + 3m gap pattern
    - Test: edge lines are solid (continuous) along full edge length
    - Test: marking width is ~0.15m
    - Test: generate_junction_surfaces for 4 approach endpoints produces a convex hull polygon at y=+0.005
    - Test: RoadSurfaceVertex size is 28 bytes (position [f32;3] + color [f32;4])
  </behavior>
  <action>
Create `crates/velos-gpu/src/road_surface.rs`:

1. `RoadSurfaceVertex` struct: `#[repr(C)] Pod/Zeroable`, fields `position: [f32;3]`, `color: [f32;4]` (28 bytes).

2. Constants: `LANE_WIDTH: f64 = 3.5`, `ROAD_COLOR: [f32;4] = [0.251, 0.251, 0.251, 1.0]` (#404040), `JUNCTION_COLOR: [f32;4] = [0.314, 0.314, 0.314, 1.0]` (#505050), `MARKING_COLOR: [f32;4] = [1.0, 1.0, 1.0, 0.8]`, `MARKING_WIDTH: f64 = 0.15`, `DASH_LENGTH: f64 = 3.0`, `GAP_LENGTH: f64 = 3.0`, `ROAD_Y: f32 = 0.0`, `MARKING_Y: f32 = 0.01`, `JUNCTION_Y: f32 = 0.005`.

3. `generate_road_mesh(graph: &RoadGraph) -> Vec<RoadSurfaceVertex>`:
   - Iterate all edges via `graph.inner().edge_weights()`
   - For each edge: expand geometry polyline to polygon using perpendicular offset (half_width = lane_count * LANE_WIDTH / 2.0)
   - For each segment pair: create 2 triangles (quad) with road color at y=ROAD_Y
   - Coordinate mapping: 2D (x, y) -> 3D (x, 0, y)
   - Use the edge_to_road_polygon pattern from RESEARCH.md

4. `generate_lane_markings(graph: &RoadGraph) -> Vec<RoadSurfaceVertex>`:
   - For each edge with lane_count >= 2: generate center dashed line (3m on, 3m off)
   - For each edge: generate left and right solid edge lines
   - Markings are thin quads (MARKING_WIDTH) at y=MARKING_Y
   - Center lines: offset at lane boundaries (i * LANE_WIDTH - half_width for i in 1..lane_count)
   - Edge lines: offset at +/- half_width

5. `generate_junction_surfaces(junction_data: &HashMap<u32, JunctionData>) -> Vec<RoadSurfaceVertex>`:
   - For each junction: collect entry/exit points from turns
   - Compute convex hull using simple angle-sort (Graham scan for small point sets)
   - Triangulate hull as fan from centroid
   - Color: JUNCTION_COLOR at y=JUNCTION_Y

Write comprehensive unit tests for all geometry generation functions.
  </action>

    <automated>cargo test -p velos-gpu road_surface -x</automated>
  </verify>
  <done>Road mesh generation produces correct vertex counts and positions for test edges, lane markings have proper dash patterns and offsets, junction surfaces are convex hulls, all vertices use correct Y-up coordinate mapping</done>
</task>
